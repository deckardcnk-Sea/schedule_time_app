# 资源释放硬性规则（任务4）— 常驻执行纪律

> 本文件是三份永久资料之一（永久资料：需求文档摘要、架构文档摘要、runtime_status.json）。
> 规则目标：防止对话上下文持续膨胀、token 被历史大段源码占满、模型自动压缩信息、进程卡死。
> 完整代码一律在磁盘，对话只写**文件路径**，不粘贴完整大段源码。

## 每完成 1 个子任务后，必须按顺序执行

1. **保存**：成功保存全部代码 + 文档 + 更新 `runtime_status.json`（写入本子任务进度）。
2. **上下文清理**：除以下三份永久资料摘要外，所有过往完整代码、历史输出、旧对话详情**不再保留在当前会话上下文**：
   - 需求文档摘要（见需求文档或 HANDOFF.md）
   - 架构文档摘要（见架构文档或 HANDOFF.md）
   - `runtime_status.json` 最新进度
3. **工具调用限制**：单个子任务内，连续文件读写/代码执行等工具调用**最多 10 轮**；
   完成本轮后，主动结束所有活跃工具会话，释放调用资源。
4. **禁止持续加载大量源码进上下文**：完整代码一律读取磁盘文件，对话只写文件路径。
5. **资源释放完成输出固定短句**：
   `【✅已完成本轮任务 + 资源释放完毕，准备执行下一项子任务】`
   随后**自动开启下一个子任务，无需等待用户指令**。
6. **膨胀预防**：若检测到上下文内容持续膨胀，主动提前精简内容，预防内存升高、模型压缩、进程卡死。

## 子任务边界约定

- 一个"子任务" = 一个可独立验证的最小功能/修复单元（如"修复 X 页 Y 颜色"、"新增 Z 持久化"）。
- 不把多个不相关改动塞进同一子任务；跨模块大改动拆成多个子任务串行。
- 每子任务结束即写 `runtime_status.json` 并清空上下文（除三永久资料）。

## 永久资料存放位置

- runtime_status.json：工程根目录 `runtime_status.json`
- 需求/架构摘要：以工程内 `HANDOFF.md` 或对应文档为准（不在聊天窗口重建）
